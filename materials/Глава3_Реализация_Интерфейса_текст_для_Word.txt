ГЛАВА 3. РЕАЛИЗАЦИЯ ПОЛЬЗОВАТЕЛЬСКОГО ИНТЕРФЕЙСА СИСТЕМЫ УПРАВЛЕНИЯ ЗАДАЧАМИ

Настоящая глава посвящена практической реализации пользовательского интерфейса прототипа системы TaskPulse: от визуального оформления и переключения темы до механизмов перетаскивания, всплывающих окон и организации описания внешнего вида в отдельных файлах стилей. Материал изложен так, чтобы читатель, не являющийся программистом, мог понять общую идею каждого решения; там, где неизбежно употребляются термины из области разработки веб-страниц, они поясняются словами «на пальцах». Листинги взяты из файлов проекта (static/js, static/scss, templates) в сокращённом виде: неполные, но совпадающие по стилю и смыслу с рабочим кодом; отдельные вспомогательные функции опущены многоточием.

────────────────────────────────────────────────────────────────────────────
3.1. Тёмная тема и переключение оформления
────────────────────────────────────────────────────────────────────────────

Современный пользователь проводит у экрана много часов подряд. Возможность выбрать тёмное оформление снижает утомляемость глаз при работе в условиях слабого освещения и придаёт интерфейсу привычный для многих приложений вид. В рамках дипломного прототипа была реализована связка из двух частей: хранение выбранного режима между сеансами работы с сайтом и собственно смена цветов фона, текста, рамок и вспомогательных элементов на всех страницах.

Первая часть опирается на встроенное в браузер хранилище небольших данных (localStorage). Это можно представить как «записку на холодильнике», которую браузер прикрепляет к конкретному адресу сайта: при следующем заходе страница может прочитать записку и узнать, какой режим оформления пользователь выбирал в прошлый раз. Если пользователь ещё ни разу ничего не выбирал, программа может ориентироваться на общесистемную настройку «светлый или тёмный режим для всех программ», которую сообщает операционная система (через стандартный запрос prefers-color-scheme).

Вторая часть — визуальное переключение. На корневом элементе страницы (тег body, то есть «тело документа», на котором лежит всё видимое содержимое) выставляется служебный признак data-theme со значением light или dark. Набор правил в каскадных таблицах стилей заранее описывает, какие цвета фона, текста и линий использовать при каждом значении этого признака. Таким образом, одна и та же разметка страницы может выглядеть по-разному: достаточно сменить один признак — и подтянется другой «комплект красок». Для плавности глазу добавлен короткий класс анимации на момент смены темы.

Особая трудность — значки в формате векторной графики (SVG). Часть значков задаётся не отдельной картинкой, а фоном элемента через таблицу стилей; в тёмом режиме такие значки должны стать светлее. В прототипе для тёмной темы подгружается текст значка, в нём заменяются шестнадцатеричные коды тёмных оттенков на светлые, и результат встраивается прямо в страницу в виде длинной «внутренней» ссылки (data URI), чтобы браузер не делал лишних запросов к серверу при каждом переключении. Для обычных значков в виде тега img при тёмной теме подменяется адрес файла на заранее подготовленную светлую версию или, аналогично, встраивается перекрашенный вариант.

Ниже приведён фрагмент из файла static/js/theme.js: чтение сохранённой темы, запись признака на элементе body и сохранение в localStorage, а также краткая анимация смены (класс theme-transition снимается через таймер). Далее — фрагмент из static/scss/style.scss: объявление переменных оформления для светлой и тёмной темы и плавные переходы для выбранных блоков при наличии класса theme-transition на body.

// static/js/theme.js (фрагмент)
const STORAGE_KEY = 'tp_theme';
const THEMES = ['light', 'dark'];

function getPreferredTheme() {
    const saved = localStorage.getItem(STORAGE_KEY);
    if (saved === 'light' || saved === 'dark') return saved;
    return window.matchMedia && window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
}

function setTheme(theme, { persist } = { persist: true }) {
    const next = THEMES.includes(theme) ? theme : 'light';
    document.body.dataset.theme = next;
    if (persist) localStorage.setItem(STORAGE_KEY, next);
    // … setIconTheme(next); setPseudoElementIconTheme(next); смена значка кнопки …
    document.body.classList.add('theme-transition');
    window.clearTimeout(setTheme._t);
    setTheme._t = window.setTimeout(() => {
        document.body.classList.remove('theme-transition');
    }, 420);
}

Пояснение. getPreferredTheme читает строку из localStorage по ключу STORAGE_KEY; если записи нет, смотрит системную настройку prefers-color-scheme. setTheme проверяет, что значение одно из допустимых THEMES, записывает его в document.body.dataset.theme (в HTML это атрибут data-theme на теге body), при необходимости дублирует в localStorage. Класс theme-transition на короткое время включает плавные переходы цветов (см. SCSS ниже).

/* static/scss/style.scss — фрагмент: переменные и тёмная тема */
body {
    background: var(--tp-bg);
    color: var(--tp-text);
    --tp-bg: #{$light-light-green};
    --tp-surface: #{$light-light-green};
    --tp-text: #{$dark-gray};
    /* … остальные --tp-* … */
}

body[data-theme='dark'] {
    --tp-bg: #{$black};
    --tp-surface: #{$dark-gray};
    --tp-text: #{$light-light-green};
    /* … */
    background: $black;
    color: $light-light-green;
}

body.theme-transition,
body.theme-transition .header,
body.theme-transition .app-container,
body.theme-transition .card,
body.theme-transition .modal-content {
    transition:
        background-color 0.35s ease,
        color 0.35s ease,
        border-color 0.35s ease,
        box-shadow 0.35s ease,
        filter 0.35s ease;
}

Пояснение к SCSS. Переменные с префиксом --tp- задают палитру; селектор body[data-theme='dark'] переопределяет их для тёмного режима. Запись #{$light-light-green} — подстановка именованного цвета из файла colors при сборке таблицы стилей. Блок body.theme-transition задаёт одинаковую длительность смены фона, текста и тени у перечисленных контейнеров, пока на body висит служебный класс анимации.

Отдельно стоит пояснить механизм отложенного обновления значков при динамическом появлении новых фрагментов страницы. Когда программа дорисовывает кусок интерфейса после загрузки данных (например, заново строит доску), в этот кусок могут попасть новые картинки-значки. Чтобы не вызывать тяжёлую перекраску на каждый микрошаг, используется короткая задержка через таймер и «наблюдатель за изменениями» дерева документа (встроенный в браузер «наблюдатель за мутациями» дерева страницы): это стандартный механизм, который сообщает скрипту «внутрь страницы что-то добавили или убрали». Получив такое сообщение, скрипт планирует единичное обновление темы для значков. Так достигается баланс между отзывчивостью и производительностью на слабых компьютерах.

Класс theme-transition на короткое время накладывается на элемент body при каждой смене темы. В таблице стилей для этого класса заданы плавные переходы для цветов фона и текста у ключевых контейнеров. Смысл простой: резкая смена контраста утомляет зрение; доли секунды сглаживания делают переключение менее «дёрганым». По истечении таймера класс снимается, чтобы не накапливать лишнюю работу у браузера при последующих операциях, не связанных с темой.

Следует помнить и об ограничениях хранилища localStorage: данные привязаны к конкретному браузеру и адресу сайта, не синхронизируются сами по себе между устройствами и не предназначены для секретной информации. Для дипломного прототипа хранение выбора темы и черновика данных канбана в этом хранилище приемлемо как учебная модель «сохранить состояние без сервера». В промышленной системе выбор темы обычно дополнительно привязывают к учётной записи пользователя на сервере.

Итог по разделу. Тёмная тема в прототипе — это не отдельная «вторая верстка», а единая структура страницы плюс переключаемый набор цветов и аккуратная обработка значков. Выбор пользователя запоминается между визитами на сайт, что соответствует ожиданиям к современному интерфейсу.

────────────────────────────────────────────────────────────────────────────
3.2. Перетаскивание элементов на экране доски задач
────────────────────────────────────────────────────────────────────────────

Доска с колонками («очередь», «в работе», «готово» и т. п.) — наглядная метафора потока задач. Пользователь ожидает, что карточку можно перенести мышью или пальцем с одной колонки на другую. Технически это реализовано подключением готовой библиотеки Sortable: она «подвешивает» обработчики на списки и следит за тем, какой элемент переносится, откуда и куда его отпустили. Библиотека — это готовый набор функций, который подключают к проекту, чтобы не писать с нуля сложную логику совместимости с разными браузерами.

В конфигурации перетаскивания задаются: какие элементы считаются переносимыми (в прототипе — строки с классом item внутри списка); как называется «группа обмена», чтобы карточку можно было перетащить между разными списками одного имени группы; небольшая задержка «чувствительности» для сенсорных экранов; визуальные классы «призрака» и «подсветки» во время движения. Особо важен фильтр: некоторые области списка (например, форма добавления задачи в очереди или текст названия задачи, по которому открывается подробное окно) не должны начинать перетаскивание всей карточки. Для этого в настройках библиотеки перечислены селекторы — короткие «адреса» элементов на странице, которые исключаются из начала перетаскивания.

После того как пользователь отпустил карточку над новым списком, срабатывает функция завершения перетаскивания. Она по внутренним номерам задачи и доски находит запись в массиве данных в памяти браузера, обновляет стадию и приоритет (срочный или обычный секции), при необходимости автоматически отмечает подзадачи выполненными при переходе в колонку «Готово» и сохраняет всё в localStorage, чтобы после обновления страницы порядок и стадии не потерялись.

Ниже — сокращённый фрагмент из static/js/board_kanban.js: создание объекта Sortable для списка задач канбана. В проекте используется вызов вида new Sortable(контейнер, { … }), а не отдельная функция «create»; контейнер — это элемент DOM, внутри которого лежат карточки.

// static/js/board_kanban.js (фрагмент функции initKanbanDnD)
const sortable = new Sortable(container, {
    animation: 150,
    group: { name: 'kanban-tasks', pull: true, put: true },
    draggable: '.item',
    filter: '.form-add-task, .kanban-task-title',
    preventOnFilter: false,
    forceFallback: true,
    fallbackOnBody: true,
    fallbackTolerance: 3,
    ghostClass: 'task-dragging',
    dragClass: 'task-drag-over',
    onStart() {
        if (cardsContainer) cardsContainer.classList.add('dragging-active');
    },
    onEnd(evt) {
        if (cardsContainer) cardsContainer.classList.remove('dragging-active');
        const toList = evt.to;
        const fromList = evt.from;
        const taskId = Number(evt.item?.dataset?.taskId);
        const boardId = Number(toList.dataset.boardId);
        if (!taskId) return;
        const task = findKanbanTask(taskId, boardId);
        if (!task) return;
        task.stage = toList.dataset.stage;
        task.priority = toList.dataset.priorityKey === 'urgent' ? 'срочно' : 'обычный';
        // … обновление подзадач при переходе в «Готово» …
        saveKanbanToLocalStorage();
        renderCurrentView();
    },
    onCancel() {
        if (cardsContainer) cardsContainer.classList.remove('dragging-active');
    }
});
container.sortable = sortable;

Пояснение. new Sortable(…) подключает внешнюю библиотеку к узлу container. Поля group, draggable и filter — это её настройки: общее имя группы для переноса между списками, какие дочерние элементы считаются карточками, какие элементы не должны начинать перетаскивание. В onEnd из объекта события evt берутся списки to и from, у карточки evt.item читается dataset.taskId; по ним находится задача в памяти, меняются поля stage и priority, затем данные сохраняются и экран перерисовывается.

Дополнительно на доске реализовано перетаскивание самих досок в общем списке проектов и перестановка колонок внутри доски, а в табличном представлении — перенос строк; везде используется та же общая идея «библиотека + обновление данных + сохранение». Это демонстрирует единый подход к интерактивности: пользователь действует жестом, программа фиксирует результат в структурированных данных и обновляет экран.

Табличный вид задаёт дополнительные требования к перетаскиванию: строка таблицы визуально растянута на несколько колонок сетки, и при «призрачном» переносе важно не потерять ширины и отступы. В прототипе для строк таблицы включён режим с подменой внешнего вида перетаскиваемой копии (режим принудительной подмены перетаскиваемой копии в библиотеке): библиотека создаёт отдельный слой-«клон», которому программно копируются вычисленные размеры и шрифты, чтобы пользователь видел ту же строку, что и до начала движения. Для неспециалиста: это компромисс между универсальностью библиотеки и сложной вёрсткой таблицы в современных браузерах.

Данные канбана в прототипе версионируются: при изменении формата хранения вводится новый номер версии, и при несовпадении с сохранённым ранее браузер заново подгружает исходные наборы задач из файлов. Это защищает от «битого» состояния после доработок дипломного проекта и поясняет на защите, что перетаскивание опирается не только на визуальный слой, но и на осмысленную модель данных.

Итог по разделу. Перетаскивание в прототипе — это не только «красивая анимация», а связка визуального действия пользователя с обновлением модели задач и сохранением в браузере. Исключения в фильтре защищают от случайного переноса при работе с формой и ссылками внутри карточки.

────────────────────────────────────────────────────────────────────────────
3.3. Всплывающие окна и сценарии работы с задачами
────────────────────────────────────────────────────────────────────────────

Многие действия удобнее выполнять во всплывающем окне поверх затемнённой страницы: так пользователь не теряет контекст (доска или список остаётся «за туманом»), но фокус внимания переносится на форму. В прототипе выделены несколько сценариев: создание новой задачи, просмотр и редактирование карточки выбранной задачи, выбор диапазона дат во всплывающем календаре, профиль пользователя, настройки. Общая часть у всех сценариев — слой overlay (перекрытие): на весь экран накладывается полупрозрачный фон с лёгким размытием, по центру показывается блок с заголовком, основным содержимым и, при необходимости, кнопками внизу.

Закрытие окна реализовано несколькими привычными способами: кнопка с крестиком, щелчок по затемнённой области вне окна, клавиша, которая в программах обычно означает «отмена текущего режима» (в технической документации её часто обозначают латинскими буквами Esc). Для слабовидящих пользователей на диалог вешаются роли и подписи в стандарте доступности (aria-*): кратко говоря, это «подсказки для программы чтения с экрана», что перед пользователем именно диалог, а не обычный текст страницы.

Календарь выбора периода дат вынесен в отдельный сценарий: при нажатии на поле даты открывается своё всплывающее окно с двумя месяцами, выделением начала и конца диапазона и кнопками «сбросить» и «применить». Так пользователь не зависит от встроенного в браузер календаря, который выглядит по-разному на разных устройствах, и получает единообразный интерфейс, совпадающий с остальным стилем системы.

Фрагмент разметки из templates/components/create_task_modal.html — слой перекрытия и диалог «Создание задачи» (начало файла; тело формы с полями опущено многоточием):

<div class="modal-overlay" id="createTaskModal" aria-hidden="true">

        <div class="modal-body create-task-modal__body">
            <!-- … поля формы: проект, название, описание, сроки и т.д. … -->
        </div>

        </div>

Пояснение. modal-overlay — полноэкранный слой; в static/scss/components/_modal.scss для него задано полупрозрачное затемнение и центрирование содержимого; класс show переключает отображение. modal-content — блок диалога с фиксированными отступами шапки и подвала. aria-hidden="true" в исходной разметке означает, что до открытия окно скрыто от вспомогательных средств; role="dialog" и aria-labelledby связывают диалог с заголовком для чтения с экрана.

Карточка задачи на доске в отдельном примере оформлена как широкое окно с тремя колонками фиксированной доли ширины окна: слева описание и общие сведения, в центре статус и сроки, справа вложения и зависимости. Высота блока с текстом описания подстраивается под колонку метаданных — это пример, как вёрстка сочетается с удобочитаемостью.

Форма создания задачи демонстрирует связку нескольких типов полей: текстовое название, многострочное описание, раскрывающиеся списки проекта и приоритета, числовое поле оценки трудозатрат, пара полей для дат с открытием общего календаря, зона для вложений и блок зависимостей с подсказками. С точки зрения пользователя всё выглядит как единый лист; с точки зрения разработки каждый тип поля имеет свои правила проверки и доступности. Например, обязательные поля помечены значком «звёздочка» и атрибутом разметки «поле обязательно для заполнения»: браузер при попытке отправить пустую форму может показать стандартное сообщение, а скрипт дополнительно может перехватить отправление для показа своего текста ошибки.

Календарный сценарий построен как отдельная функция, которую можно вызвать из разных мест интерфейса. На вход она получает уже выбранные ранее даты (если есть) и функцию обратного вызова, которая выполнится после нажатия кнопки «Применить». На выходе пользователь видит два месяца подряд, может щёлкать по дням, задавать начало и конец диапазона, сбрасывать выбор и подтверждать. Такой подход уменьшает дублирование кода между фильтрами на странице списка задач, формой в очереди канбана и карточкой задачи.

Итог по разделу. Всплывающие окна в прототипе унифицированы по внешнему виду и по способам закрытия; формы создания и просмотра задач опираются на те же элементы ввода, что и остальной интерфейс; календарь диапазона дат выделен в отдельный повторно используемый сценарий.

────────────────────────────────────────────────────────────────────────────
3.4. Главная страница и наглядные сводки для пользователя
────────────────────────────────────────────────────────────────────────────

Стартовая страница системы решает сразу несколько задач: напомнить о ближайших сроках, показать календарные события, дать быстрый обзор загрузки проектов и состава команды, отобразить динамику активности и последние изменения по задачам. Такой «пульт управления» снижает необходимость переходить вглубь разделов для рутинной проверки ситуации.

Блок «Задачи к выполнению» построен как простая таблица из строк: слева название задачи и подпись проекта, справа дата. Цвет даты варьируется в зависимости от срочности (например, более яркий оттенок для самых близких сроков). Тексты сформулированы на русском языке без ненужных заимствований из английского, чтобы демонстрационные данные выглядели естественно в отчётной документации и на защите.

Блок «Ближайшие события» использует ту же сетку колонок для единообразия. Перечень событий включает как организационные пункты (завершение итерации, разбор итогов работы команды), так и учебно-практические (практикум по строгой типизации при разработке для обозревателя).

Блок «Активные проекты» на главной странице использует горизонтальные полосы долей: зелёный сегмент показывает долю завершённых работ, более светлый — долю находящихся в работе, серый — не начатые задачи. Подписи к цветам вынесены в легенду под полосами, чтобы пользователю не нужно было угадывать значение цвета. Такой приём относится к категории «сводной диаграммы без осей»: он хорошо читается на плакате защиты и не требует от зрителя знания терминов статистики.

Блок «Команда» сочетает фотографии участников и короткие подписи должностей. Статус «в сети» показан маленьким индикатором у части сотрудников: это упрощённая модель присутствия, достаточная для дипломного интерфейса и понятная без пояснительной записки.

Блок «Последние действия» имитирует ленту обновлений: слева портрет, рядом номер и название задачи, далее цветной ярлык состояния, справа дата и время. Сетка колонок согласована с блоком задач к выполнению, чтобы визуальный ритм страницы не «ломался» при чтении сверху вниз.

Блок активности команды содержит векторную диаграмму (формат SVG): это описание рисунка прямо в разметке, которое масштабируется без «мыла» при изменении ширины окна. Скрипт на языке JavaScript подставляет координаты линий и подписи по осям из подготовленных массивов чисел. Для неспециалиста: массив — это пронумерованный список значений в памяти программы; скрипт берёт очередное значение, переводит его в координату на экране и соединяет точки линией.

Фрагмент из static/js/index.js: функция drawMiniChart строит подписи по оси X и ломаные линии (polyline) в слое linesLayer внутри SVG-элемента miniChart на главной странице.

// static/js/index.js (фрагмент drawMiniChart)
function drawMiniChart(team, me, opts) {
    const workingDays = getLastWorkingDays();
    const xLabels = workingDays.map((date, index) => formatLabel(date, index, workingDays.length));
    opts = opts || {};
    const minV = ('min' in opts) ? opts.min : 2;
    const maxV = ('max' in opts) ? opts.max : 14;
    const normalize = (arr) => {
        const out = Array.isArray(arr) ? arr.slice(0, 5) : [];
        while (out.length < 5) out.push(minV);
        return out;
    };
    const teamA = normalize(team);
    const meA = normalize(me);
    const xs = [];
    for (let i = 0; i < 5; i++) xs.push(left + i * stepX);
    xLabelsGroup.innerHTML = '';
    for (let i = 0; i < 5; i++) {
        const t = document.createElementNS('http://www.w3.org/2000/svg', 'text');
        t.setAttribute('x', xs[i]);
        t.setAttribute('y', 320);
        t.setAttribute('text-anchor', 'middle');
        t.textContent = xLabels[i] || '';
        xLabelsGroup.appendChild(t);
    }
    const teamYs = teamA.map(v => computeY(v, minV, maxV));
    const meYs = meA.map(v => computeY(v, minV, maxV));
    linesLayer.innerHTML = '';
    const makePolyline = (xs, ys, cls) => {
        const poly = document.createElementNS('http://www.w3.org/2000/svg', 'polyline');
        poly.setAttribute('points', xs.map((x, i) => `${x},${ys[i]}`).join(' '));
        poly.setAttribute('class', cls);
        linesLayer.appendChild(poly);
    };
    makePolyline(xs, teamYs, 'mc-line-team');
    makePolyline(xs, meYs, 'mc-line-me');
    // … отрисовка круговых маркеров в pointsLayer …
}

Пояснение. getLastWorkingDays собирает пять последних рабочих дней; normalize приводит массивы team и me к длине пять. computeY переводит числовое значение активности в координату Y внутри заранее заданной области графика. document.createElementNS с пространством имён SVG создаёт узлы text и polyline не как обычный HTML, а как элементы векторного рисунка; атрибут points задаёт последовательность точек ломаной линии в виде «x,y x,y …». Классы mc-line-team и mc-line-me задают разный цвет линий в таблице стилей.

Итог по разделу. Главная страница спроектирована как набор карточек с повторяющейся сеткой и общей типографикой (гарнитура, кегль, интерлиньяж), что создаёт ощущение цельного продукта, а не набора случайных блоков.

────────────────────────────────────────────────────────────────────────────
3.5. Организация стилей и повторное использование оформления
────────────────────────────────────────────────────────────────────────────

По мере роста проекта хранить все правила оформления в одном гигантском файле неудобно: трудно искать, выше риск случайно сломать несвязанный экран. В прототипе применён препроцессор Sass (расширение синтаксиса каскадных таблиц стилей): исходные тексты разнесены по папкам components, pages, util, а главный файл style.scss только перечисляет подключения директивой @import в нужном порядке. Порядок важен: сначала базовые цвета и шрифты, затем компоненты, затем особенности конкретных страниц — чтобы более частные правила могли при необходимости уточнять общие.

Повторяющиеся оформления полей ввода, раскрывающихся списков и многострочных полей вынесены в общие фрагменты (миксины и шаблоны %placeholder в терминологии Sass). Миксин — это как «заготовка абзаца» в текстовом редакторе: один раз описали набор свойств, затем вставляете заготовку в десять мест; при правке заготовки обновляются все вставки. Так достигается визуальная согласованность модальных окон фильтрации задач, создания задачи и карточки задачи.

Для темы используются переменные с префиксом --tp- на корневом элементе или на body: они задают, например, цвет фона страницы и цвет «поверхности» карточки. Переключение тёмной темы сводится к подстановке другого набора значений этих переменных.

Фрагмент начала главного файла static/scss/style.scss — порядок подключения модулей (часть директив опущена):

@import 'colors';
@import 'mixins';
@import 'typo';
@import 'variables';
@import 'components/header';
@import 'components/modal';
@import 'components/calendar';
@import 'components/forms/controls';
@import 'components/modals/create_task';
@import 'components/modals/profile';
@import 'components/modals/settings';
@import 'pages/auth';
@import 'pages/boards';
@import 'pages/task_detail_static';
/* … далее общие правила и сброс отступов … */

Пояснение. Директива @import в Sass при сборке подставляет содержимое указанного файла. Сначала подключаются цвета, миксины и переменные, затем компоненты интерфейса (шапка, модальные окна, календарь, общие элементы форм), затем отдельные модальные сценарии и страницы (доски, карточка задачи). Так порядок импорта задаёт каскад приоритетов: страничные правила могут уточнять общие.

Итог по разделу. Разбиение стилей и выделение общих примитивов уменьшают дублирование, упрощают согласование светлой и тёмной темы и делают правки локальными: чаще всего меняется один небольшой файл, а не вся таблица стилей целиком.

Отдельно в проекте предусмотрен вспомогательный файл стилей для печати: при выводе на принтер можно скрыть навигацию и декоративные элементы, оставив только содержательные таблицы и заголовки. Хотя печать в эпоху экранов используется реже, наличие такого файла показывает зрелость подхода к оформлению «на все случаи жизни» и хорошо смотрится в пояснительной записке как пример инженерной аккуратности.

────────────────────────────────────────────────────────────────────────────
3.6. Связь реализованного интерфейса с ожиданиями пользователя и ограничениями прототипа
────────────────────────────────────────────────────────────────────────────

Сопоставим кратко содержание главы с типичными ожиданиями пользователя системы управления задачами. Во-первых, пользователь хочет видеть актуальные сроки и состояния: главная страница и доска закрывают этот запрос за счёт списков, цветовых акцентов дат и перетаскивания между колонками. Во-вторых, пользователь хочет меньше рутинных щелчков: всплывающие окна сосредоточивают внимание на одной операции (создать задачу, уточнить срок) и уменьшают потерю контекста. В-третьих, пользователь ожидает предсказуемого внешнего вида: модульность стилей и переменные темы помогают держать интерфейс цельным при расширении функций.

Ограничения дипломного прототипа следует проговорить открыто. Данные хранятся преимущественно в браузере пользователя, без полноценного сервера приложений: это упрощает демонстрацию на защите, но не отражает вопросы одновременной работы нескольких человек над одной доской, разграничения прав и резервного копирования. Часть сценариев (например, загрузка вложений) визуально показана, но не связана с реальным файловым хранилищем. Эти моменты не умаляют ценности главы как описания пользовательского интерфейса, но задают рамку для главы о перспективах внедрения.

────────────────────────────────────────────────────────────────────────────
Заключение по главе 3
────────────────────────────────────────────────────────────────────────────

В главе рассмотрены ключевые стороны реализации пользовательского интерфейса системы TaskPulse: переключаемая тёмная тема с запоминанием выбора, перетаскивание карточек и других элементов доски с обновлением данных, унифицированные всплывающие окна для работы с задачами и календарём, компоновка главной страницы с наглядными сводками и диаграммой, а также модульная организация таблиц стилей с повторным использованием оформления. Показано, что пользовательский интерфейс опирается на сочетание разметки страницы, правил внешнего вида и сценариев на языке JavaScript, причём каждый из этих слоёв решает свою часть задачи по отношению к удобству и предсказуемости работы пользователя.

Отдельно подчеркнём методическую линию изложения: для каждого крупного решения даны бытовые аналогии (хранилище браузера как записка, переменные стилей как именованные ячейки, библиотека перетаскивания как готовый набор правил совместимости), а фрагменты кода служат иллюстрацией логики, а не учебником синтаксиса. Такой баланс между технической точностью и доступностью текста соответствует задаче дипломной работы — показать не только факт реализации, но и понимание сути принятых проектных решений.

Практическая ценность главы для продолжения работы над проектом состоит в том, что выделены узлы расширения: при появлении серверной части достаточно заменить источник данных и место сохранения, сохранив отработанные в прототипе визуальные шаблоны; при появлении новых экранов — подключить новый файл стилей и переиспользовать миксины полей ввода; при ужесточении требований доступности — усилить разметку диалогов и сценарии управления с клавиатуры, не переписывая вёрстку с нуля. Таким образом, реализованный интерфейс выполняет роль устойчивого каркаса для дальнейшего развития системы.
